
# Как пользоваться якорными файлами WebLoadTester

**Версия:** v1.1 02.02.2026

## 1. Что такое «якорные файлы»
Якорные файлы — это *канон документации* проекта WebLoadTester: требования, архитектура, UI-спецификация, карта кода, глоссарий, план испытаний и матрица трассируемости.

Их задача:
- быть единым «источником истины» для тебя и для ChatGPT/Codex;
- снять двусмысленность: что мы делаем, почему и как это связано;
- ускорять реализацию: в документации фиксируются контракты, сущности, экраны, данные и связи.

## 2. Порядок внесения изменений (обязательный регламент)
1) **Сначала** правим `01_КАНОН_ПРОЕКТА_И_ТЗ...` (решения/границы/критерии).
2) **Потом** синхронизируем:
   - `02_ТЕХ_СПЕЦИФИКАЦИЯ...` (архитектура, контракты, БД, отчёты),
   - `03_UI_СПЕЦИФИКАЦИЯ...` (экраны/UX/валидация),
   - `04_КАРТА_КОДА_И_СВЯЗИ...` (файлы/классы/потоки данных),
   - `07_МАТРИЦА_ТРАССИРУЕМОСТИ...` (требование → реализация → тест).
3) В конце фиксируем изменения в `08_CODEX_ПРОМПТЫ_И_ЖУРНАЛ_ИЗМЕНЕНИЙ...`.

## 3. Как использовать в работе с Codex
- Перед большим изменением кода прикладывай *релевантные* якорные файлы (обычно 01–04 + 08).
- В промпте явно указывай: «Следуй канону якорных файлов. Если есть конфликт — приоритет у 01, затем 02, затем 03».
- После PR проверяй: сборка, запуск, UI, миграции БД/файлы артефактов, корректность путей.

## 4. Нумерация версий
- Любое изменение якорного файла → увеличиваем версию (`v1.1`, `v1.2`, …) и оставляем дату.
- Внутри текстов **не** размножаем «Версия vX.Y» по разделам; версию держим в шапке.

## 5. Быстрый чек-лист консистентности
- Везде: **3 семейства модулей, 10 модулей, 4 вкладки верхнего уровня (3 семейства + Прогоны)**.
- Нагрузка: **только Iterations/Duration** в ядре; stress/soak — только в перспективах.
- Telegram: **опционально**; ошибки Telegram **не влияют** на статус прогона.
- Отчёты: JSON — всегда; HTML — опционально.
- RunId: каждый прогон имеет уникальный RunId и папку `runs/<runId>/`.
